# AI 뉴스레터 자동화 서비스 기획서

## 1. 서비스 개요

본 서비스는 AI 관련 국내외 정보를 자동으로 수집·정리하고, 이를 회사 내부 구성원에게 이메일 뉴스레터 형태로 제공하는 사내 AI 인텔리전스 자동화 시스템이다.

콘텐츠는 하나의 뉴스 묶음으로 제공하지 않고, 다음 두 개의 트랙으로 구분한다.

1. **AI 활용 전문가가 볼 만한 최신 AI 뉴스**
2. **일반 임직원이 실제로 활용할 수 있는 AI 활용 인사이트**

이를 통해 AI 담당자, 기획자, 개발자, 전략 담당자에게는 빠른 기술·산업 동향을 제공하고, 일반 임직원에게는 실무에 바로 적용 가능한 AI 활용법을 제공한다.

---

## 2. 핵심 목표

### 2.1 전문가용 최신 AI 뉴스

AI 기술, 산업, 정책, 연구, 제품 업데이트를 빠르게 파악할 수 있도록 정리한다.

주요 대상은 다음과 같다.

- AI 담당자
- 개발자
- 기획자
- 전략 담당자
- 기술 리더
- 디지털 전환 담당자

주요 목적은 다음과 같다.

- 최신 AI 기술 동향 파악
- 빅테크 및 스타트업 동향 확인
- 정책·규제·저작권 이슈 추적
- 회사 전략 및 업무에 미칠 영향 분석
- 후속 관찰이 필요한 이슈 식별

---

### 2.2 일반 임직원용 AI 활용 인사이트

일반 회사 구성원이 실제 업무에서 AI를 사용할 수 있도록 실용적인 활용법을 제공한다.

주요 대상은 다음과 같다.

- 일반 사무직
- 영업
- 마케팅
- 기획
- 관리부서
- 기술·설계 실무자
- 보고서·문서 작성 업무가 많은 직원

주요 목적은 다음과 같다.

- 업무 생산성 향상
- 문서 작성, 자료 조사, 회의록 정리, 보고서 작성 지원
- 바로 복사해 쓸 수 있는 프롬프트 제공
- AI 사용 시 보안·개인정보·검증 리스크 안내
- 부서별 활용 사례 제공

## 3. 기본 운영 원칙

### 3.1 자동화하되 최종 발송은 사람이 검수한다

초기 버전에서는 완전 자동 발송을 하지 않는다.

권장 흐름은 다음과 같다.

```text
자동 수집
→ 자동 정리
→ LLM 초안 생성
→ 관리자 검수
→ 승인 후 이메일 발송
```

뉴스레터는 회사 내부 커뮤니케이션이므로 오요약, 과장 표현, 민감한 표현, 잘못된 출처 사용을 방지하기 위해 **Human-in-the-loop** 방식을 사용한다.

---

### 3.2 LLM은 선별된 데이터에만 사용한다

LLM은 모든 수집 데이터에 무작정 사용하지 않는다.  
비용, 속도, 토큰 사용량을 고려해 코드 기반 처리로 먼저 데이터를 줄인 뒤, 가치가 높은 후보에만 LLM을 사용한다.

기본 원칙은 다음과 같다.

```text
많이 수집한다
→ 코드로 최대한 정리하고 줄인다
→ 중요한 후보만 LLM에 보낸다
→ 짧은 입력으로 요약한다
→ 최종 뉴스레터 작성에 고성능 모델을 사용한다
```

---

### 3.3 초기에는 과도한 자동화 구조를 피한다

MVP 단계에서는 복잡한 에이전트 시스템, GraphRAG, 대규모 크롤링 시스템을 사용하지 않는다.

권장 발전 순서는 다음과 같다.

| 단계 | 구조 |
|---|---|
| MVP | DB + 규칙 기반 필터 + LLM 요약 |
| 1차 고도화 | 임베딩 검색, 중복 클러스터링, 중요도 점수 |
| 2차 고도화 | 회사 관심사 기반 RAG |
| 장기 고도화 | Knowledge Graph, GraphRAG, 트렌드 분석 |

## 4. 전체 자동화 프로세스

자동화 프로세스는 다음 5단계로 구분한다.

```text
1. 데이터 수집
2. 데이터 처리
3. 데이터 통합
4. 뉴스레터 작성
5. 배포
```

## 6. 데이터 처리 단계

### 6.1 목적

수집한 데이터를 뉴스레터에 사용할 수 있는 표준 형태로 정리한다.

---

### 6.2 처리 항목

| 처리 항목 | 설명 | 자동화 방식 |
|---|---|---|
| 정규화 | 제목, URL, 출처, 날짜, 언어, 요약 필드 통일 | 코드 |
| 언어 감지 | 한국어/영어/일본어 등 구분 | 코드 또는 경량 모델 |
| 중복 제거 | 같은 기사, 같은 원문, 같은 보도자료 제거 | 코드 + 선택적 LLM |
| 관련도 필터링 | AI와 무관한 기사 제거 | 키워드 + LLM 보조 |
| 카테고리 태깅 | 모델, 정책, 투자, 연구, 활용법 등 분류 | 규칙 + LLM |
| 신뢰도 평가 | 출처 등급, 원문 여부, 인용 여부 평가 | 코드 + 정책 |
| 위험 필터링 | 루머, 광고성, 출처 불명 콘텐츠 제외 | 코드 + LLM 보조 |
| 트랙 분류 | 전문가용 뉴스 / 실무 활용 인사이트 분류 | 코드 + LLM |

## 7. 데이터 통합 단계

### 7.1 목적

여러 소스에서 들어온 정보를 하나의 뉴스레터 후보군으로 통합하되, 최종적으로는 두 개의 콘텐츠 트랙으로 나눈다.

---

### 7.2 데이터 흐름

```text
수집
→ 정규화
→ 중복 제거
→ Track 분류
   ├─ expert_news
   │   ├─ 기술/산업 중요도 평가
   │   ├─ 클러스터링
   │   └─ 전문가용 뉴스 작성
   │
   └─ practical_insight
       ├─ 업무 활용 가능성 평가
       ├─ 활용 시나리오 생성
       ├─ 프롬프트 예시 생성
       └─ 일반 직원용 인사이트 작성
→ 통합 뉴스레터 초안
→ 관리자 검수
→ 이메일 배포
```

---

### 7.3 통합 기능

| 기능 | 설명 |
|---|---|
| Topic Clustering | 같은 이슈를 하나의 주제로 묶음 |
| Source Merging | 같은 주제에 대해 국내/해외/공식 출처 연결 |
| Track Classification | 전문가용 뉴스와 일반 활용 인사이트 구분 |
| Priority Scoring | 회사에 중요한 순서로 점수화 |
| Freshness Scoring | 최신성 반영 |
| Impact Scoring | 산업·기술·업무 영향도 평가 |
| Candidate Selection | 뉴스레터 후보 선정 |

## 9. 배포 단계

### 9.1 기본 배포 방식

MVP에서는 이메일 중심으로 운영한다.

| 방식 | MVP 여부 | 설명 |
|---|---:|---|
| 이메일 초안 생성 | 포함 | 관리자 검수용 |
| 이메일 자동 발송 | 제외 | 초기에는 승인 후 발송 |
| Slack 발송 | 확장 후보 | 요약 카드 형태 |
| Microsoft Teams 발송 | 확장 후보 | 조직 환경에 따라 선택 |
| Notion 저장 | 확장 후보 | 아카이브용 |
| 사내 게시판 업로드 | 확장 후보 | 장기 운영 시 고려 |

---

### 9.2 승인 후 발송 원칙

초기 버전에서는 다음 상태 흐름을 따른다.

```text
collected
→ processed
→ candidate
→ drafted
→ review_required
→ approved
→ sent
```

`approved` 상태가 되기 전에는 자동 발송하지 않는다.

## 10. 버티컬 슬라이스 아키텍처

### 10.1 기본 방향

전통적인 레이어 구조보다 기능 단위로 슬라이스를 나눈다.

권장 구조는 다음과 같다.

```text
source-management/
collection/
processing/
integration/
newsletter/
distribution/
monitoring/
admin-review/
```

각 슬라이스는 자기 기능에 필요한 API, 서비스, 저장소, 프롬프트, 테스트를 내부에 가진다.

---

### 10.2 슬라이스 구성

| 슬라이스 | 책임 |
|---|---|
| Source Management | 수집 소스 등록, 수정, 삭제, 활성화, 테스트 |
| Collection | 네이버 뉴스, RSS, 유튜브 RSS 등에서 데이터 수집 |
| Processing | 정규화, 중복 제거, 필터링, 태깅 |
| Integration | 주제 묶기, 중요도 평가, 후보 선정 |
| LLM Prompt Management | 프롬프트 버전 관리, 용도별 템플릿 관리 |
| Newsletter | 뉴스레터 초안 생성, 섹션 구성 |
| Admin Review | 관리자 검토, 수정, 승인 |
| Distribution | 이메일 발송, Slack/Teams 확장 |
| Archive | 발송 이력, 원문 링크, 요약 결과 저장 |
| Monitoring | 실패 로그, 비용, 토큰 사용량, 품질 지표 |

## 11. Source Registry 설계

### 11.1 목적

사용자가 데이터 수집 소스를 직접 지정, 추가, 삭제할 수 있도록 별도의 목록으로 관리한다.

단순 URL 목록이 아니라 **Source Registry**로 관리한다.

---

### 11.2 Source Registry 필드

| 필드 | 설명 | 예시 |
|---|---|---|
| source_id | 고유 ID | openai-blog |
| name | 소스명 | OpenAI Blog |
| type | 수집 방식 | NAVER_API / RSS / YOUTUBE_RSS / API / MANUAL |
| content_track | 콘텐츠 트랙 | expert_news / practical_insight / both |
| url | RSS URL 또는 API Endpoint | RSS URL |
| query | 검색어 | AI, LLM, automation |
| language | 언어 | ko, en |
| region | 지역 | KR, US, Global |
| category | 카테고리 | AI Model, Prompting, Productivity |
| audience_level | 독자 난이도 | beginner / intermediate / expert |
| priority | 우선순위 | high / medium / low |
| trust_level | 신뢰도 | official / media / community |
| enabled | 활성 여부 | true |
| fetch_interval | 수집 주기 | hourly / daily / weekly |
| auth_required | 인증 필요 여부 | true / false |
| rate_limit_note | 제한 사항 | API 호출량 제한 등 |
| owner | 관리 담당자 | admin |
| created_at | 생성일 | YYYY-MM-DD |
| updated_at | 수정일 | YYYY-MM-DD |

## 13. LLM Prompt Management

### 13.1 목적

LLM을 사용하는 단계의 프롬프트를 코드 안에 직접 넣지 않고 별도로 관리한다.

권장 구조는 다음과 같다.

```text
prompts/
  expert-news/
    expert-relevance-filter.md
    expert-category-classifier.md
    expert-importance-scorer.md
    expert-cluster-summarizer.md
    expert-news-writer.md

  practical-insight/
    practical-relevance-filter.md
    practical-usecase-generator.md
    practical-prompt-example-generator.md
    practical-risk-checker.md
    practical-insight-writer.md

  common/
    source-grounding-checker.md
    hallucination-checker.md
    newsletter-editor.md
```

## 22. CLI 개발 방향

CLI에서는 각 단계를 독립적으로 실행할 수 있도록 설계한다.

```text
newsletter sources:list
newsletter sources:add
newsletter sources:disable

newsletter collect --date today
newsletter process --date today
newsletter classify-track --date today

newsletter draft --date today --track expert-news
newsletter draft --date today --track practical-insight
newsletter draft --date today --track all

newsletter review --date today
newsletter send --issue latest

newsletter run --date today --until draft
```

핵심은 `run` 하나에 모든 것을 넣지 않고 다음 단계별 실행이 가능하게 만드는 것이다.

```text
collect
process
integrate
draft
review
send
```

이렇게 나누면 실패한 단계만 재실행할 수 있다.

## 26. 최종 서비스 정의

본 서비스는 다음과 같이 정의한다.

> **AI 관련 정보를 국내 뉴스, 해외 RSS, 유튜브 RSS 등에서 수집하고, 이를 `전문가용 최신 AI 뉴스`와 `일반 임직원용 AI 활용 인사이트`의 두 트랙으로 분류한 뒤, LLM을 활용해 각각의 독자 수준에 맞는 요약·분석·활용 콘텐츠를 생성하고, 관리자가 검수한 후 이메일 뉴스레터로 배포하는 사내 AI 인텔리전스 뉴스레터 자동화 시스템.**

## 27. 최종 권장 구조

```text
모듈러 모놀리스
+ 버티컬 슬라이스 아키텍처
+ Source Registry
+ RSS/API 중심 수집
+ 코드 기반 전처리
+ 제한적 LLM 처리
+ 전문가용 뉴스 / 일반 활용 인사이트 2트랙 구성
+ Human-in-the-loop 검수
+ 이메일 중심 배포
```

초기에는 **자동 발송 시스템**이 아니라 **자동 초안 생성 + 검수 발송 시스템**으로 구축한다.  
이 방향이 비용, 정확도, 운영 리스크, 개발 난이도 측면에서 가장 균형이 좋다.
